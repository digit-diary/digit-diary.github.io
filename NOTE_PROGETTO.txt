================================================================================
  PANORAMICA
================================================================================

  Applicazione web single-file (index.html) per la gestione del personale
  del Casino Lugano SA. Permette di registrare eventi, redigere rapporti
  giornalieri, inviare note tra colleghi, generare moduli disciplinari
  con firma digitale e QR code, consultare statistiche, gestire costi
  Maison con budget e grafici, assegnare promemoria con scadenze,
  scrivere consegne turno, generare report settimanali PDF, e gestire
  visibilita di pagine/funzioni per operatore. Include una Dashboard
  panoramica come homepage con KPI in tempo reale.

  Tecnologie utilizzate:
  - HTML/CSS/JavaScript (tutto in un unico file)
  - Supabase REST API + RPC autenticate (database PostgreSQL cloud)
  - Supabase Realtime (WebSocket per note istantanee)
  - Supabase JS Client v2 (per Realtime)
  - Web Crypto API: AES-256-GCM (crittografia note), PBKDF2 (password)
  - RLS (Row Level Security) su tutte le tabelle
  - jsPDF + AutoTable (esportazione PDF)
  - Chart.js (grafici statistiche + scheda collaboratore)
  - Flatpickr (calendario date)
  - Groq API (AI: llama-3.3-70b-versatile per testo,
    llama-3.2-90b-vision-preview per immagini)
  - qrcode-generator (QR code su PDF moduli)

  - Password: PBKDF2 con 100K iterazioni e salt per utente
  - Auto-migrazione: login con SHA-256 → upgrade a PBKDF2, cancella v1
  - Rate limiting: max 5 tentativi/minuto per utente (server-side)
  - Admin session tokens: RPC admin (add/remove operator, force change pwd,
    set groq key) richiedono token valido
  - Sessione admin validata al reload, invalidata al logout (24h scadenza)
  - Credenziali Supabase offuscate (XOR, non in chiaro nel sorgente)
  - Chiave Groq API cifrata XOR nel DB, bloccata da RLS REST
  - RLS deny_all_anon su TUTTE le tabelle (non solo sensibili)
  - Tutte le operazioni via RPC server-side (SECURITY DEFINER)
  - pwd_hash, pwd_hash_v2, recovery_code, groq_api_key bloccati da RLS
  - admin_sessions, operator_sessions, login_attempts non accessibili via REST
  - Token sessione per tutti i flussi di login (password, biometrico, refresh)
  - upsert_impostazione RPC per salvataggio affidabile impostazioni
  - SRI (Subresource Integrity) su tutti i 10 script/CSS CDN
  - Rate limiting anche su change_op_pwd, recovery_master_pwd
  - Push notifications: API secret (x-push-secret) + CORS ristretto
  - push_subscriptions: RLS zero-access (solo service_role_key via Edge Function)
  - WebAuthn (Face ID/Touch ID): credenziale locale, verifica biometrica ad ogni rientro
  - Sessione basata su timestamp localStorage (8h senza biometrico, illimitata con)
